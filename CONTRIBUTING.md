# TOKKI 기여 가이드 (CONTRIBUTING)

이 문서는 TOKKI 프로젝트에 기여하는 모든 팀원을 위한 작업 기준서입니다.
복잡한 용어보다 "바로 실행 가능한 규칙" 중심으로 정리했습니다.

## 1) 기여 시작 전 필수 체크

- `README.md`, `CODE_OF_CONDUCT.md`, 본 문서를 먼저 읽습니다.
- 작업 전 해당 기능/버그에 대한 이슈가 있는지 확인합니다.
- 이슈가 없다면 간단한 이슈를 먼저 생성하고 목적/범위를 명확히 합니다.
- 작업 범위가 큰 경우, 바로 코딩하지 말고 접근 방향을 먼저 합의합니다.

## 2) 브랜치 전략

기본 흐름은 문서/위키와 동일하게 `dev` 통합 + `main` 안정 브랜치 모델을 따릅니다.

권장 브랜치 네이밍:

- `feat/<short-topic>`: 신규 기능
- `fix/<short-topic>`: 버그 수정
- `docs/<short-topic>`: 문서 수정
- `refactor/<short-topic>`: 리팩터링

예시:

- `feat/auth-login`
- `fix/calendar-null-error`
- `docs/erd-update`

머지 규칙:

- 기능 개발은 `feat/*`, `fix/*` 등 작업 브랜치에서 시작해 `dev`로 PR 생성
- `dev` 브랜치는 통합 브랜치로 운영
- `main` 브랜치는 안정 버전만 유지하며 직접 push 금지
- `dev -> main` 반영은 팀 합의 기준(위키 기준) 충족 시 진행

## 3) 커밋 메시지 규칙

커밋은 작고 명확하게 나눕니다. 한 커밋에는 하나의 목적만 담습니다.

형식:

`type: 한글 요약`

권장 type:

- `feat`: 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 변경
- `refactor`: 리팩터링
- `test`: 테스트 추가/수정
- `chore`: 기타 설정/정리

예시:

- `feat: 회원가입 유효성 검사 추가`
- `fix: 일정 등록 시 시간대 오류 수정`
- `docs: 클래스 다이어그램 최신화`

## 4) Pull Request(PR) 규칙

PR은 팀원이 "빠르게 이해하고 검토"할 수 있어야 합니다.

PR 본문에 아래 내용을 포함해주세요.

1. 변경 목적 (왜 했는지)
2. 주요 변경 사항 (무엇이 바뀌었는지)
3. 테스트 방법 (어떻게 확인했는지)
4. 스크린샷/영상 (UI 변경 시 필수)
5. 관련 이슈 (예: `Closes #12`)

권장 크기:

- 리뷰 가능한 단위(가능하면 300줄 내외)
- 너무 큰 PR은 분할해서 올립니다.

## 5) 코드 리뷰 문화

리뷰의 목적은 "사람 평가"가 아니라 "코드 품질 개선"입니다.

- 리뷰어: 문제 지적 + 개선 방향을 함께 제시
- 작성자: 방어적 반응보다 근거 기반 설명
- 논쟁이 길어지면 1:1 또는 짧은 회의로 빠르게 정리
- 승인 전 강제 수정사항은 명확히 구분 (`must` / `nit`)

### 리뷰 SLA (팀 권장)

- PR 생성 후 24시간 이내 1차 리뷰
- 피드백 반영 후 24시간 이내 재리뷰
- 일정 지연 예상 시 PR/채팅에 미리 공유

## 6) 테스트 및 품질 기준

최소 기준:

- 본인이 변경한 기능은 직접 실행/검증
- 기존 기능에 회귀(regression)가 없는지 확인
- 에러 로그/경고가 없는 상태로 PR 제출

가능하면 아래도 포함:

- 단위 테스트 또는 수동 테스트 시나리오 기록
- 실패 케이스(엣지 케이스) 확인 결과

## 7) 문서화 기준

다음에 해당하면 문서를 함께 업데이트합니다.

- 요구사항 변경
- API/DB 스키마 변경
- 화면 흐름/시퀀스 변경
- 팀 규칙 변경

업데이트 대상 예시:

- `ERD.md`
- `Class_Diagram.md`
- `Sequence_Diagram.md`
- `요구사항명세서.md`
- `decisions.md`

## 8) 이슈 작성 규칙 (템플릿/라벨)

프로젝트 관리 문서 기준으로, 아래 이슈 템플릿을 사용합니다.

- `Bug Report`
- `Documentation`
- `Meeting / Decision Record`
- `PBI / User story`
- `Refactor / Tech debt`
- `Spike / Research`
- `Feature`

이슈 생성 시 라벨도 함께 지정합니다. 자주 쓰는 라벨:

- `Backend`, `Fullstack`, `Design`, `test`
- `priority/high`, `priority/medium`, `priority/low`
- `needs-triage`, `Blocked`, `question`, `help wanted`
- `pbi`, `sprint`, `task`, `refactor`, `spike`, `Decision`

이슈 제목:

`[type] 짧은 문제/요청 설명`

이슈 본문:

- 배경: 왜 필요한가?
- 목표: 무엇을 달성해야 하나?
- 범위: 어디까지 할 것인가?
- 완료 조건(DoD): 무엇이 되면 끝인가?
- 참고자료: 링크, 캡처, 문서

## 9) Definition of Done (완료 기준)

아래 항목을 모두 만족하면 "완료"로 봅니다.

- 요구사항 충족
- 코드 리뷰 반영 완료
- 테스트 완료 및 결과 확인
- 문서 업데이트 완료(필요 시)
- 관련 이슈/PR 링크 정리 완료

## 10) 커뮤니케이션 원칙

- 진행 상황은 숨기지 말고 짧게라도 공유
- 일정 리스크는 조기에 알리고 대안 제시
- 모르는 내용은 추측 대신 질문
- 개인 사정으로 지연 시 사전 공유

## 11) 첫 기여자를 위한 체크리스트

- [ ] 작업 이슈를 만들거나 할당받았다.
- [ ] 브랜치를 규칙에 맞게 생성했다.
- [ ] 변경 내용을 로컬에서 확인했다.
- [ ] PR 본문을 템플릿에 맞게 작성했다.
- [ ] 리뷰 피드백을 반영했다.
- [ ] 필요한 문서를 업데이트했다.

---

질문이 있거나 규칙 개선이 필요하면, 이 문서에 PR로 제안해주세요.
TOKKI 팀은 "빠르게 만들기"보다 "지속 가능하게 협업하기"를 더 중요하게 생각합니다.
